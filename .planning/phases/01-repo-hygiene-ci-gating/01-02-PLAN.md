    Create `.github/workflows/ci.yml` with the following exact contents. Preserve
    indentation (2 spaces), key order, and quoting precisely — GitHub Actions YAML
    is sensitive to shape and the acceptance checks grep on specific tokens.

    ```yaml
    name: CI

    on:
      pull_request:
        branches: [main]
      push:
        branches: [main]

    permissions:
      contents: read

    concurrency:
      group: ci-${{ github.ref }}
      cancel-in-progress: true

    jobs:
      build:
        name: build
        runs-on: ubuntu-latest
        timeout-minutes: 10

        steps:
          - name: Checkout
            uses: actions/checkout@v5

          - name: Set up Node (pinned via .nvmrc)
            uses: actions/setup-node@v5
            with:
              node-version-file: '.nvmrc'
              cache: 'npm'

          - name: Install dependencies
            run: npm ci

          - name: Build site
            run: npm run build

          - name: Sanity-check build output
            run: |
              test -d dist
              test -f dist/index.html
              test -f dist/404.html
    ```

    Rationale for each block (do NOT include rationale in the file — it is inline
    here for the executor):
    - `on.pull_request.branches: [main]` — gate PRs targeting `main` (CI-01).
    - `on.push.branches: [main]` — also run on push-to-main so the green check on
      `main` itself stays fresh and serves as a sanity ribbon for the default
      branch. This does NOT create a merge block loop because a push to `main`
      happens only after a merged PR already passed the same check.
    - `permissions: contents: read` — principle of least privilege. No secrets,
      no write scope, no `pull_request_target` (which would expose the token to
      forked-PR code). See `<threat_model>` below.
    - `concurrency` — cancel superseded runs when a PR branch is force-pushed.
    - `timeout-minutes: 10` — generous upper bound; a current `npm ci` + `astro
      build` completes in a minute or two on `ubuntu-latest`.
    - `actions/setup-node@v5` with `node-version-file: '.nvmrc'` — single source
      of truth for Node version. A future `.nvmrc` bump propagates automatically
      and keeps CI, Cloudflare Pages (which also reads `.nvmrc`), and local dev
      in lockstep.
    - `cache: 'npm'` — setup-node's built-in npm cache keyed off
      `package-lock.json` hash. Shaves ~30s off subsequent runs.
    - `npm ci` (not `npm install`) — honors the lockfile exactly; matches the
      command documented in README.md (from Plan 01) and CONTRIBUTING.md.
    - Final sanity step — asserts the build produced the two files that must
      always exist (`dist/index.html` from `src/pages/index.astro`, and
      `dist/404.html` which Astro emits automatically). This catches the rare
      "build succeeded but emitted nothing" failure mode cheaply.

    Constraints:
    - Do NOT use `pull_request_target` — it would give the workflow access to
      repo secrets even when triggered by a forked PR, which is the canonical
      GitHub Actions script-injection footgun.
    - Do NOT add a `pull_request_review` or `schedule` trigger — out of scope.
    - Do NOT add secrets, env vars, or `NODE_ENV` overrides — Astro builds
      deterministically from the lockfile.
    - Do NOT add `continue-on-error: true` on any step — this is a gate; failures
      must block.
    - Pin action versions to major tags (`@v5`) to match the style already used
      in `new-release.yml`. Commit-SHA pinning is stricter but creates upgrade
      noise the maintainer has to track manually; major-tag pinning is
      acceptable for this low-risk static-site build.
    - Do NOT modify `.github/workflows/new-release.yml`.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner ↔ repository | The workflow runs with `GITHUB_TOKEN` which, by default in a public-repo `pull_request` event, is read-only but still scoped to the repo. |
| Forked-PR contributor → workflow | If the repo ever accepts PRs from forks, a forked contributor controls the checked-out source code (including `package.json` scripts and the Astro config). |
| `.nvmrc` → runner | The workflow reads Node version from `.nvmrc`. A malicious PR could theoretically change `.nvmrc` to an unavailable version to deny-of-service the build, but that would merely fail the check — which is exactly what the gate is designed to do. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-02-01 | Tampering | `.github/workflows/ci.yml` trigger | mitigate | Use `on: pull_request` (not `on: pull_request_target`). `pull_request` runs the workflow with the PR's code in a context that does NOT have access to repo secrets, which prevents forked-PR script injection from exfiltrating tokens. Explicit acceptance check: workflow file must NOT contain the string `pull_request_target`. |
| T-01-02-02 | Elevation of privilege | `GITHUB_TOKEN` scope | mitigate | Top-level `permissions: contents: read` clamps the token to read-only. The workflow does not push, comment, or label. Acceptance check: workflow must contain `contents: read` and must NOT contain `contents: write`. |
| T-01-02-03 | Tampering | Action versions (`@v5` tags) | accept | Major-tag pinning trusts the action author not to ship a malicious `v5.x.y` under the same major. For this low-risk static build, the maintenance burden of SHA pinning outweighs the incremental safety; `actions/checkout` and `actions/setup-node` are first-party GitHub actions with a strong track record. If threat model shifts (e.g. handling secrets, publishing artifacts), revisit and pin to commit SHAs. |
| T-01-02-04 | Denial of service | Concurrency group | mitigate | `concurrency.cancel-in-progress: true` on `ci-${{ github.ref }}` prevents a contributor from stacking dozens of runs by force-pushing to the same branch. |
| T-01-02-05 | Information disclosure | Workflow logs | accept | No secrets are referenced anywhere in the workflow, so logs cannot leak them. The build's stdout (Astro messages, file paths) is already public information for a static site. |
| T-01-02-06 | Repudiation | Required status check bypass | accept | Task 2 does NOT force-check "Include administrators". The solo maintainer retains the ability to bypass the check in an emergency. This is a deliberate tradeoff for a solo-developer workflow; flipping that checkbox is a one-click operation if the team later grows. |
| T-01-02-07 | Tampering | Cloudflare Pages deploy (out of this plan's scope) | accept | Cloudflare Pages builds on push-to-main regardless of the GitHub check status; the CI gate here prevents a broken build from reaching `main` in the first place, which indirectly protects the Pages deploy. A direct Pages-side gate is out of scope (Pages project settings live outside this repo). |
</threat_model>
