## Decisions Made

- **Theme names: `night-owl` / `night-owl-light` (Shiki-bundled) instead of the BRIEF-specified `starlight-dark` / `starlight-light`.** The latter are Starlight-exclusive aliases and are not resolvable from the standalone `astro-expressive-code` package — the build crashed with `bundledThemes[bundledThemeName] is not a function` on the first run. Starlight maps `starlight-dark`/`starlight-light` to Night Owl variants internally (confirmed in `@astrojs/starlight/integrations/expressive-code/theming.ts`), so the Shiki-bundled `night-owl` / `night-owl-light` give near-identical visuals on `/posts/*` while leaving Starlight's `/docs/*` theming untouched.

- **Two Expressive Code integrations coexist (not "Starlight defers to standalone").** BRIEF line 102 and CONTEXT line 51 claimed Starlight v0.38.3 auto-detects and defers to a standalone install. Reading Starlight's `integrations/expressive-code/index.ts` showed this is not the case — Starlight always calls `astroExpressiveCode(...)` with its own config preprocessor regardless of whether a user-level standalone instance exists. Both instances run side-by-side: Starlight's wrapper owns `/docs/*` (with `starlight-dark`/`starlight-light` themes), the standalone owns `/posts/*` (with `night-owl`/`night-owl-light`). Verified no collision: `/docs/getting-started/index.html` still ships `class="expressive-code"` markup and `ec-line` syntax-highlighted spans after the standalone integration was added.

- **Schema stays inline in content.config.ts** (matching the existing single-collection pattern), not extracted to a schema module. CONTEXT.md left this to Claude's discretion — inline keeps the file single-source for the two collections we have now.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Theme names `starlight-dark` / `starlight-light` are Starlight-exclusive and crash the standalone Expressive Code build**
- **Found during:** Task 3 build verification (first `npm run build` after all three plan tasks had been applied).
- **Issue:** The plan, BRIEF, and CONTEXT all specified `themes: ['starlight-dark', 'starlight-light']` for the standalone `astro-expressive-code` integration. These theme names only exist inside Starlight's Expressive Code wrapper (they are defined at `@astrojs/starlight/integrations/expressive-code/theming.ts:5` as `BundledThemeName = 'starlight-dark' | 'starlight-light'` and mapped to Starlight's own `night-owl-dark.jsonc` / `night-owl-light.jsonc`). The standalone `astro-expressive-code` package only knows Shiki-bundled theme names. Running `npm run build` with the BRIEF-specified config failed with `[astro-expressive-code] bundledThemes[bundledThemeName] is not a function`.
- **Fix:** Changed `themes` to `['night-owl', 'night-owl-light']` — Shiki's bundled equivalents of the themes Starlight renames. Verified via Context7 docs ("Starlight Integration Options … Provides two themes (`starlight-dark` and `starlight-light`) … specific to Starlight") and by reading Starlight's own theming source.
- **Files modified:** `astro.config.mjs` (one line: theme-array literal).
- **Verification:** `npm run build` exits 0 cleanly. `dist/docs/getting-started/index.html` still contains `class="expressive-code"` wrappers and `ec-line` syntax-highlighted spans, proving Starlight's `/docs/*` rendering is unaffected by the standalone integration.
- **Committed in:** `45c30bb` (separate `fix(02.1-01):` commit, intentionally not folded into the Task 2 wire-up commit so bisect can isolate the theme-name change).

### Related finding (no code change)

**BRIEF/CONTEXT claim "Starlight auto-defers to standalone Expressive Code" is not accurate for Starlight 0.38.3.** Starlight always calls `astroExpressiveCode(...)` with its own config preprocessor regardless of whether a user-level standalone integration is registered (see `@astrojs/starlight/integrations/expressive-code/index.ts:108`). The two integrations coexist — they do NOT merge or share theme registries. The integration-ordering rule ("standalone first") still holds as defensive config hygiene and remains future-compatible if Starlight ever implements true deferral, but the user-facing effect today is "two EC instances render separately, one per docs-vs-non-docs markdown surface." Documented here so Wave 5 (CF Pages preview verification) doesn't re-discover it as a surprise.
