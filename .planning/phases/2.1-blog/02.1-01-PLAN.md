<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Repo dependency tree → Cloudflare Pages build | Two new npm packages are pulled during `npm ci` on the CF build runner and executed at build time. Supply-chain compromise of either package could execute arbitrary code during build. |
| `src/content/posts/` → published HTML | This plan creates the directory only (empty). Content arrives in later plans / branches. The collection's schema is the first validation gate for anything placed there. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.1-01 | Tampering | `astro-expressive-code` (new dep) | mitigate | Install via `npm install` so SHA-512 integrity hashes land in `package-lock.json`; CF Pages uses `npm ci` which enforces lockfile integrity. Pin via caret range (not wildcard). Reputable Astro-ecosystem package (maintained by Hippo/Expressive Code team). |
| T-02.1-02 | Tampering | `@astrojs/rss` (new dep) | mitigate | Same as T-02.1-01 — official Astro integration, lockfile-gated, caret-pinned. |
| T-02.1-03 | Elevation of Privilege | Expressive Code integration in `astro.config.mjs` | accept | Integrations run at build time with full FS access; both new deps are well-known in the Astro ecosystem and no `exec`/`child_process` usage is configured (we pass `themes` only, no custom `plugins`). Risk is bounded by the overall npm supply chain we already accept. |
| T-02.1-04 | Information Disclosure | `og_image` Zod field (accepts any string) | accept in this plan; revisit | Current schema is `z.string().optional()`. Threat: a future malicious committer could set `og_image: "javascript:..."` which would end up in a `<meta property="og:image">` tag. Mitigation is owned by **Plan 03 Task 2** (per-post `<meta>` emission): the executor MUST either tighten the schema with `.startsWith('/')` / `.url()` OR validate at emission time. Tracked as a follow-up; NOT blocking for Wave 1 because no posts exist yet. |
| T-02.1-05 | Denial of Service | Content collection with no content | mitigate | `.gitkeep` ensures `getCollection('posts')` returns `[]` cleanly. Without the directory, `astro build` throws "collection not found" — already mitigated by this task's Edit 2. |

Block: `T-02.1-01` and `T-02.1-02` — build fails loudly if lockfile integrity check fails on CF Pages.
Accept: `T-02.1-03`, `T-02.1-04` (deferred to Plan 03), `T-02.1-05` (mitigated by `.gitkeep`).
</threat_model>
