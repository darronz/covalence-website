    Reviewer must complete ALL of the following in a real browser on the Cloudflare Pages preview URL for the feature branch (URL pattern: `https://<commit-hash>.<project>.pages.dev/` — confirm the exact URL via the PR's CF Pages status check or the Cloudflare dashboard → Pages → covalence-website-v2 → Deployments).

    SURFACE A — Homepage
    1. Visit `<preview-url>/`. Confirm the page loads fully.
    2. Inspect `<head>`: confirm `<link rel="alternate" type="application/rss+xml" title="Covalence Blog" href="/posts/rss.xml">` is present.
    3. Check the Nav — primary nav row should show exactly: "How It Works | Features | Releases | Blog | Docs". Click "Blog" — should navigate to `<preview-url>/posts/`.
    4. Shrink the viewport below 900px so the hamburger appears. Open the drawer. Confirm it lists: "How It Works | Features | Releases | Blog | Docs" in the same order. Click "Blog" — should navigate to `<preview-url>/posts/`.
    5. Scroll to the Footer — confirm there is a small "RSS" link. Click it — should load the feed at `<preview-url>/posts/rss.xml`.
    6. LANDING BAND: if the collection is empty (`.gitkeep` only), confirm NO "Latest writing" section appears on the homepage. If the collection has at least one post, confirm the band appears between the Architecture section and the Footer, shows up to 2 posts (cards with title, date, description), and has a "Read all posts →" link pointing at `/posts/`.

    SURFACE B — Blog index `/posts/`
    7. Visit `<preview-url>/posts/`. Confirm the page renders with marketing-site chrome: Nav at top, page-header ("Writing" eyebrow + "Blog" H1 + lede sentence), list area, Footer.
    8. If collection empty: the page shows only the header. If collection has posts: reverse-chronological list of cards (title, ISO-formatted `en-GB` date, description) is shown.
    9. Inspect `<head>`: confirm `<link rel="alternate" type="application/rss+xml">` still present (site-wide coverage).
    10. View source and confirm the page does NOT use Starlight chrome — no sidebar, no Starlight header bar. Pure marketing-site typography.

    SURFACE C — Per-post page `/posts/<slug>/` (only if a test post exists on the branch)
    11. Visit a `<preview-url>/posts/<slug>/` URL. If none exists, SKIP this surface and note in resume-signal: "No test post on branch; per-post surface unverified."
    12. Confirm: Nav at top, title block (H1 = post title, meta line with date and author), horizontal rule, body content, Footer.
    13. Inspect `<head>`: confirm `<meta property="og:type" content="article">`, `<meta property="article:published_time" content="...">`, `<meta property="article:author" content="...">`, per-post `<link rel="canonical" href="https://covalence.app/posts/<slug>/">` are all present (from Plan 03's head-extra slot via Plan 04's Base.astro slot exposure).
    14. Confirm body CSS: H2 / H3 / paragraph / list / blockquote / link styling from `posts.css` applies visibly. Compare informally against the Figma/reference if available.
    15. If the post has ```code blocks```: confirm they render with Expressive Code chrome — syntax colouring, copy button at top-right of block, frame/filename header (if the fenced block has a `title` attribute).

    SURFACE D — RSS feed `/posts/rss.xml`
    16. Visit `<preview-url>/posts/rss.xml` directly in the browser. Confirm it renders as XML (not a 404, not a redirect, not HTML-escaped).
    17. View source. Confirm: `<?xml version="1.0" ...>` on line 1; `<rss ...>` root with `<channel>` containing `<title>Covalence Blog</title>`; if posts exist, one `<item>` per post with `<title>`, `<description>`, `<pubDate>`, `<link>https://covalence.app/posts/<slug>/</link>` (MUST be on covalence.app, NOT `*.pages.dev`), and `<content:encoded>` carrying rendered HTML.
    18. Paste the preview RSS URL into https://validator.w3.org/feed/ (or any RSS validator). Confirm zero errors. Warnings about relative image URLs are acceptable.
    19. OPTIONAL: add the feed to a feed reader (NetNewsWire / Feedly / Reeder). Confirm it parses and the post body renders.

    SURFACE E — /docs/* code blocks (CRITICAL — the collision-risk check)
    20. Visit `<preview-url>/docs/getting-started/`. Locate any code block on the page.
    21. Confirm the code block has SYNTAX COLOURING (not plain black-on-white or monochrome monospace). This proves Expressive Code is actually processing `/docs/*` content after the standalone integration registration.
    22. Hover over the code block: confirm a "Copy" button appears in the top-right.
    23. Compare visually against a `/posts/<slug>/` code block (if a test post with code exists) — they should be stylistically identical (same theme, same padding, same frame chrome).
    24. If the docs code block is PLAIN / UNSTYLED: STOP. The Starlight-vs-standalone collision has regressed. Note this in resume-signal; gap-closure plan will need to address ordering in `astro.config.mjs` or Starlight config overrides.

    SURFACE F — Mobile / narrow viewport spot-checks
    25. Resize to ~375px width. Confirm: Nav collapses to hamburger, drawer opens/closes smoothly, Blog link is visible and tappable, Footer RSS link is readable, `/posts/` page header shrinks gracefully (`.page-header h1` font-size drops per the 768px media query).

    SURFACE G — Build output / deploy metadata
    26. Via Cloudflare Pages dashboard: confirm the preview deploy status is "Success". If "Build failed" or "Deployment failed", STOP and report the CF build log error.
    27. Confirm `_routes.json` exclusions still apply — try visiting `<preview-url>/releases/Covalence-latest.dmg` (should still redirect / serve as before; the blog phase did NOT edit routing exclusions).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CF Pages preview URL → reviewer's browser | Standard HTTPS from Cloudflare. No secrets in preview builds (site is fully static; no API keys, no auth). |
| Reviewer's browser → external RSS validator | Reviewer pastes a public preview URL into a public validator. URL itself is not secret. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.1-19 | Information Disclosure | Preview URL pasted into W3C feed validator | accept | Preview URLs are public by design (CF Pages default). No secrets disclosed. |
| T-02.1-20 | Repudiation | Human approval for merge | mitigate | Resume-signal is recorded in the session log; the PR merge commit ties back to the approving reviewer's identity (git commit author + PR approver). |

Block: none — this plan makes no code changes; it is a human-in-the-loop gate.
Accept: both threats above — standard static-site preview-review flow.
</threat_model>
