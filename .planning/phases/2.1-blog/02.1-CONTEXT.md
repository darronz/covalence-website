<domain>
## Phase Boundary

Ship a `/posts/` content collection alongside the existing marketing site and Starlight docs.

**In scope:**
- Astro content collection at `src/content/posts/` (plain markdown, Zod-validated frontmatter).
- Bespoke routes: `src/pages/posts/index.astro` (list), `src/pages/posts/[...slug].astro` (detail), `src/pages/posts/rss.xml.ts` (RSS 2.0 feed).
- `astro-expressive-code` installed as a standalone integration so `/docs/*` and `/posts/*` share one code-block config; Starlight auto-defers to it.
- Two new components: `PostList.astro` (reused on index + home-page band) and `PostLayout.astro` (per-post wrapper).
- Hand-authored prose CSS scoped under `.prose` in `src/styles/posts.css`.
- Nav gains a "Blog" link between "Releases" and "Docs" on desktop + mobile drawer. Footer gains a small "RSS" link. `Base.astro` advertises the feed via `<link rel="alternate" type="application/rss+xml">`.
- Landing-page "Latest writing" band between `Architecture` and `Footer`, gated by a single `showLatestWriting` boolean in `src/pages/index.astro`. Renders nothing if the collection is empty (no placeholder).

**Out of scope for v1 (locked ŌĆö see `<deferred>` below):**
- Tags, categories, pagination, reading time, draft flag, per-post OG image generation, MDX, author pages, in-post search, related-posts, comments.

</domain>

<decisions>
## Implementation Decisions

### URL & naming
- URL segment is `/posts/`; nav label is "Blog". (Searchable word in nav, clean URL slug.)
- Files named `YYYY-MM-DD-<kebab-slug>.md` in `src/content/posts/`. Date prefix is filing ergonomics only ŌĆö published URL is `/posts/<slug>/` derived from frontmatter `slug` OR filename minus the date prefix.
- `[...slug].astro` (rest param) used intentionally so future nesting (e.g. `/posts/deep-dives/xyz`) doesn't require a route migration.

### Content collection
- New collection `posts` in `src/content.config.ts`, sibling to the existing docs collection. Zod schema:

  | Field          | Type     | Required | Purpose |
  |----------------|----------|----------|---------|
  | `title`        | string   | yes      | H1 + `<title>` + OG |
  | `description`  | string   | yes      | meta/OG/RSS; <160 chars |
  | `pub_date`     | Date     | yes      | ordering + display |
  | `updated_date` | Date     | no       | optional freshness marker |
  | `author`       | string   | no       | default "Darron Schall" |
  | `og_image`     | string   | no       | absolute `/public` path; fall back to site default |

  Explicitly NOT in the schema for v1: `draft`, `tags`, `reading_time`, `slug` (slug derived, not user-specified).

### Code blocks
- `astro-expressive-code` added as a standalone integration in `astro.config.mjs`.
- Themes: `starlight-dark` / `starlight-light` (match docs code-block visuals).
- Starlight (v0.38.3) bundles Expressive Code internally; per Starlight docs, a standalone install supersedes it. **Landmine**: verify in preview build that `/docs/*` code blocks still render correctly after the integration lands ŌĆö collision risk must be checked, not assumed.

### Prose styling
- Hand-authored CSS in `src/styles/posts.css`, scoped under a `.prose` wrapper.
- Target elements: `<h2>`, `<h3>`, `<p>`, `<ul>`, `<ol>`, `<blockquote>`, `<a>`, `<hr>`, `<img>`. Nothing else.
- Code blocks are **Expressive Code's concern**, not `posts.css`. Do not double-style.
- Expect styling to need tuning after the first real post is written ŌĆö ship v1 against the Project Context Sync explainer content, not against Lorem Ipsum.

### Components
- `src/components/PostList.astro` ŌĆö takes `posts: CollectionEntry<'posts'>[]` and optional `limit?: number`. Used by both the index page and the home-page "Latest writing" band. Renders title + date + description + link per post.
- `src/components/PostLayout.astro` ŌĆö wraps the post body in: title block, pub-date, author, `<hr>`, prose container. Uses `Base.astro` for nav/footer.
- Reused unchanged (except for declared edits): `Nav.astro`, `Footer.astro`, `Base.astro`.

### SEO / metadata (per post)
- `<title>`, `<meta description>`, `<link rel="canonical">` (absolute URL on `covalence.app`).
- Open Graph: `og:title`, `og:description`, `og:type=article`, `og:url`, `og:image`.
- Twitter card: summary_large_image with matching fields.
- Article metadata: `article:published_time`, `article:author`.
- **No** JSON-LD `Article` schema in v1.
- Sitemap / robots / site-wide canonical are **explicitly deferred to Phase 3** ŌĆö the phase is sequenced before Phase 3 precisely so Phase 3 picks up `/posts/*` in the same sweep.

### Home-page integration
- "Latest writing" band inserted in `src/pages/index.astro` between `Architecture` and `Footer`.
- Renders 2 most recent posts via `<PostList posts={latestPosts} limit={2} />`.
- Empty collection ŌåÆ render nothing. No "coming soon" placeholder.
- Gated by a single boolean `showLatestWriting` at the top of `index.astro` so the landing page can A/B without touching components.

### Navigation
- `Nav.astro` desktop link row AND mobile drawer both gain a `<a href="/posts/">Blog</a>` entry, placed between "Releases" and "Docs".
- `Footer.astro` gains a small "RSS" text link pointing at `/posts/rss.xml`.
- `Base.astro` gets `<link rel="alternate" type="application/rss+xml" title="Covalence Blog" href="/posts/rss.xml">` in the head tag on every page.

### RSS feed
- `src/pages/posts/rss.xml.ts` using `@astrojs/rss`.
- RSS 2.0, full rendered HTML body in `<content:encoded>` (not just description).
- Capped at 50 items. (Bursty cadence ŌĆö 50 is far beyond current pace; revisit if archive grows past that.)

### Voice / prose constraints (impacts fixture content + default copy in components)
- British spelling.
- Short bursts, no ceremony, allergic to superlatives.
- **Do NOT force "blog post shape"** on the layout ŌĆö no big TL;DR blocks, summary bands, or CTAs at the end of posts.

### Claude's Discretion
- Exact DOM structure of `PostList.astro` and `PostLayout.astro` (within the constraints above).
- Specific CSS values in `posts.css` (font sizes, spacing, link colour) ŌĆö must feel continuous with `global.css` tokens; may need iteration post-first-post.
- Whether the Blog landing-page band uses the component's full layout or a trimmed variant. Either is fine as long as the underlying data access is shared.
- Whether frontmatter Zod schema lives inline in `content.config.ts` (matching the existing docs collection pattern) or is extracted.
- Exact test strategy: phase currently has no test infra; build-gate + manual CF Pages preview is the v1 verification floor.

</decisions>

## Deferred Ideas

**Explicitly locked OUT of v1 (not oversights ŌĆö deliberately excluded):**

- Tags, categories, `/posts/tag/[tag]/` routes (premature until archive Ōēź ~20 posts).
- Reading time, word count.

- Comments, reactions, per-post analytics.
- In-post search (Starlight search doesn't see `/posts/*`; not adding a second search stack).
- "Related posts" section.
- Pagination on the index page (reverse-chron list only).
- MDX (plain markdown only; opt-in per-post if a concrete embed need appears).
- JSON-LD `Article` structured data (can be added alongside other SEO work in Phase 3 if needed).
- Sitemap / `robots.txt` / site-wide canonical ŌĆö covered by Phase 3's SEO sweep precisely because Phase 2.1 is sequenced before Phase 3.
