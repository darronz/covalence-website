# Phase 2.1 Plan 08: Gap-Closure — Hide Blog Nav + RSS Footer Link When Posts Collection Empty Summary

**Two shared .astro components (Nav + Footer) gained frontmatter blocks that compute `const hasPosts = (await getCollection('posts')).length > 0` at build time, and three visible anchors (Nav desktop Blog, Nav drawer Blog, Footer RSS) got wrapped in `{hasPosts && …}` expressions — with the posts collection currently empty (`.gitkeep` only), every page that renders these components now hides its `/posts/*` entry-points; the instant the first post lands, the entries reappear on the next build with zero config change. Head-level `<link rel="alternate" type="application/rss+xml">` metadata in both Base.astro and the Starlight head-injection config stays unconditional (invariant-preserving, invisible to users, auto-useful for crawlers on first-post publish).**

## Decisions Made

All captured in the `key-decisions` frontmatter block. The three material ones:

1. **Gate at the anchor level, not at the Nav/Footer block level** — Other links must stay; only `/posts/*` entry-points hide.
2. **Keep head-level `<link rel="alternate">` unconditional** — Invisible metadata, auto-useful for crawlers, breaks nothing when feed is empty (RSS 2.0 allows empty channels).
3. **Preserve existing trailing-slash paths in Nav.astro** — The plan's `<interfaces>` block showed `/releases` / `/docs` without slashes; that was target-shape guidance, not a literal patch. Touching those would have violated the plan's own "exactly 2 files, 3 wrappers, nothing else" contract.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Task 3 Part B `RSS_HOME -eq 0` assertion was over-strict; Task 3 acceptance_criteria `grep -c 'href="/posts/rss.xml"' == 0` was internally self-contradictory with Part C1 `RSS_META_HOME == 1`**

- **Found during:** Task 1 baseline capture — `grep -oE 'href="/posts/rss\.xml"' dist/index.html | wc -l` returned `2` in the pre-fix baseline, which was surprising (Footer only has one `<a>` link). Inspection showed the `<link rel="alternate" type="application/rss+xml" href="/posts/rss.xml">` head tag from Base.astro (Wave 4) also contains the same `href="/posts/rss.xml"` substring. The pattern conflates `<a>` anchors with `<link>` metadata.
- **Issue:** Plan Task 3 Part B asserted `[ "$RSS_HOME" -eq 0 ]` after the fix, which would necessarily fail because the preserved Wave 4 `<link rel="alternate">` still contains `href="/posts/rss.xml"`. Part C1 of the same task explicitly requires that link to remain (`RSS_META_HOME == 1`). Both cannot be simultaneously true. The plan's intent is clear from the must-haves ("no RSS anchor in the Footer region" — anchor, not URL occurrence), so the assertion pattern needed to be precise.
- **Fix:** Used `grep -oE '<a [^>]*href="/posts/rss\.xml"' ...` (anchor-tag-only) for the Task 3 Part B assertion. This counts only visible `<a>` elements, not `<link>` metadata. The result: pre-fix baseline = 1 (Footer anchor only), post-fix = 0 (anchor gated). Also recorded the URL-wide count (`href="/posts/rss.xml"` anywhere) as a secondary baseline (`rss-url-refs-home.txt`) so the invariant delta (2 → 1) is explicit in the metrics table.
- **Files modified:** None — this was a verification-logic fix, not a source-code fix. No additional commits.
- **Verification:** Post-fix homepage `grep -oE '<a [^>]*href="/posts/rss\.xml"' dist/index.html | wc -l` = 0 ✓; `grep -oE 'rel="alternate"[^>]+type="application/rss\+xml"' dist/index.html | wc -l` = 1 ✓ (Wave 4 invariant preserved).
