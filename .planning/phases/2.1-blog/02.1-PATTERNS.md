### `src/pages/posts/[...slug].astro` (new) — per-post detail route

**Analog:** None internal. Starlight owns `/docs/*` via its integration; there is no hand-rolled dynamic content-collection route in this repo. This is greenfield but **Astro-standard** — not freestyle.

**Standard Astro pattern** the executor must follow (from Astro content collections docs — no codebase analog to crib from):

```astro
---
import { getCollection, render } from 'astro:content';
import type { CollectionEntry } from 'astro:content';
import Base from '../../layouts/Base.astro';
import Nav from '../../components/Nav.astro';
import Footer from '../../components/Footer.astro';
import PostLayout from '../../components/PostLayout.astro';

export async function getStaticPaths() {
  const posts = await getCollection('posts');
  return posts.map((post) => ({
    params: { slug: post.id },
    props: { post },
  }));
}

interface Props {
  post: CollectionEntry<'posts'>;
}

const { post } = Astro.props;
const { Content } = await render(post);
---

<Base title={`${post.data.title} — Covalence`} description={post.data.description}>
  <Nav />
  <PostLayout post={post}>
    <Content />
  </PostLayout>
  <Footer />
</Base>
```

**Props typing** — follow `src/layouts/Base.astro` lines 5-10 convention: local `interface Props`, destructure from `Astro.props`.

**Per-post SEO head tags** — BRIEF.md requires `canonical`, OG article tags, Twitter card, `article:published_time`, `article:author`. `Base.astro` currently only exposes `title` and `description` props. The executor has two options (planner decides):
1. Extend `Base.astro`'s Props with optional `ogType`, `publishedTime`, `author`, `ogImage` fields and branch in the head.
2. Emit the additional `<meta>` tags via an Astro `<head>` slot injection from within `[...slug].astro`.

The existing `<meta>` pattern to mirror is `src/layouts/Base.astro` lines 25-38 (grouped OG / Twitter blocks).

### `src/pages/posts/rss.xml.ts` (new) — RSS 2.0 feed endpoint

**Analog: NONE.** This is the only file in the phase with **no internal reference** in the repo. The repo has:
- Zero `.ts` route endpoints (only `src/content.config.ts`, which is not a route).
- Zero uses of `@astrojs/rss` (the dep itself is being added in this phase).

The executor must work from `@astrojs/rss` docs directly. Flag this to the planner as **no internal pattern — library docs are the source of truth**.

**Minimal shape the executor should land on** (from `@astrojs/rss` docs — provided here so planner doesn't have to):

```typescript
import rss from '@astrojs/rss';
import { getCollection } from 'astro:content';
import { marked } from 'marked';
import type { APIContext } from 'astro';

export async function GET(context: APIContext) {
  const posts = (await getCollection('posts'))
    .sort((a, b) => b.data.pub_date.valueOf() - a.data.pub_date.valueOf())
    .slice(0, 50);

  return rss({
    title: 'Covalence Blog',
    description: 'Long-form writing on Covalence.',
    site: context.site!,
    items: posts.map((post) => ({
      title: post.data.title,
      description: post.data.description,
      pubDate: post.data.pub_date,
      link: `/posts/${post.id}/`,
      // Full-body HTML in <content:encoded> per BRIEF — @astrojs/rss handles encoding.
      content: /* rendered HTML */ '',
    })),
  });
}
```

**Rendering full body HTML** — `@astrojs/rss` expects a string for `content`. Options: (a) use `marked.parse(post.body)` (the `marked` dep is already present, used in `src/pages/releases.astro` line 6-8), or (b) use Astro's `render()` + container API. **`marked` is already wired in this repo** — preferring it keeps the dependency surface flat.

**Style match** — `releases.astro` line 6-8 shows the project's existing `marked` usage:

```typescript
import { marked } from 'marked';
marked.setOptions({ gfm: true });
```

That init block should be shared or duplicated for the RSS file.

### `src/content.config.ts` (edit) — add `posts` collection

**Analog:** The **same file**, lines 1-10 (the existing `docs` entry):

```typescript
import { defineCollection } from 'astro:content';
import { docsLoader } from '@astrojs/starlight/loaders';
import { docsSchema } from '@astrojs/starlight/schema';

export const collections = {
  docs: defineCollection({
    loader: docsLoader(),
    schema: docsSchema(),
  }),
};
```

**Delta:** Starlight provides its own `loader` and `schema`. The `posts` collection needs neither — it uses Astro's default glob loader and a hand-rolled Zod schema.

**Shape the executor should produce** (6 fields per CONTEXT.md `<decisions>` → Content collection — 3 required, 3 optional, snake_case keys per project CONVENTIONS.md):

```typescript
import { defineCollection, z } from 'astro:content';
import { glob } from 'astro/loaders';
import { docsLoader } from '@astrojs/starlight/loaders';
import { docsSchema } from '@astrojs/starlight/schema';

export const collections = {
  docs: defineCollection({
    loader: docsLoader(),
    schema: docsSchema(),
  }),
  posts: defineCollection({
    loader: glob({ pattern: '**/*.md', base: './src/content/posts' }),
    schema: z.object({
      title: z.string(),
      description: z.string().max(160),
      pub_date: z.date(),
      updated_date: z.date().optional(),
      author: z.string().default('Darron Schall'),
      og_image: z.string().optional(),
    }),
  }),
};
```

**Note:** CONTEXT.md `<decisions>` → "frontmatter Zod schema lives inline in `content.config.ts` (matching the existing docs collection pattern) or is extracted" — either is fine; inline matches the single-file convention of the existing entry.

## Shared Patterns

These patterns cut across multiple files in this phase. Executors working on any of the new/edited files should consult them.

### Design tokens (colour / spacing / typography)

**Source:** `src/styles/global.css` lines 1-30.
**Apply to:** `posts.css`, any scoped `<style>` block in `PostList.astro` / `PostLayout.astro` / `src/pages/posts/index.astro`.
**Rule:** reference `var(--token-name)` — do NOT hardcode colour or spacing values. This is enforced by existing components (`Hero.astro`, `Features.astro`, `releases.astro`, etc. all reference tokens only).

Token set available:
- Colours: `--bg-primary`, `--bg-surface`, `--text-primary`, `--text-secondary`, `--accent`, `--accent-hover`, `--accent-glow`, `--accent-subtle`, `--accent-subtle-hover`, `--border`
- Typography: `--font-sans`, `--font-mono`
- Spacing scale: `--space-xs` (0.25rem) → `--space-2xl` (8rem)
- Layout: `--max-width` (1120px), `--nav-height` (64px)

### Page shell composition

**Source:** `src/pages/releases.astro` lines 42-47, 137-139; `src/pages/index.astro`.
**Apply to:** `src/pages/posts/index.astro`, `src/pages/posts/[...slug].astro`.
**Rule:** Non-docs pages always follow `<Base><Nav /><main>…</main><Footer /></Base>`. `Base` gets `title` and `description` props. `Nav` and `Footer` are zero-prop.

### Scoped component CSS

**Source:** every `.astro` in `src/components/`.
**Apply to:** `PostList.astro`, `PostLayout.astro`.
**Rule:** single `<style>` block at the bottom. Astro auto-scopes. Mobile-last breakpoints: `@media (max-width: 768px)`, `@media (max-width: 640px)`, `@media (max-width: 1024px)`. No `<script>` tags (the nav hamburger is the only component in the repo with client JS — don't add more without reason).

### Build-time data access

**Two established patterns.** Pick the right one per file.

1. **JSON import** (build-time data from `src/data/*.json`) — used by `Hero.astro` line 2, `releases.astro` line 5. Direct ES-module import. Appropriate for release metadata.
2. **Content collection** (`getCollection`) — the **new pattern** this phase introduces. All blog data access uses this. Appropriate for markdown-backed content.

**Do not mix.** Blog files MUST use `getCollection('posts')`. JSON-importing the collection won't work.

### File-naming conventions (CONVENTIONS.md)

- Components: `PascalCase.astro` → `PostList.astro`, `PostLayout.astro` ✓
- Pages: `lowercase.astro` matching URL segment → `posts/index.astro`, `posts/[...slug].astro` ✓

- JSON/data keys: `snake_case` → `pub_date`, `updated_date`, `og_image` ✓ (the schema field names in `content.config.ts` match this)
