## Prose CSS Scoping Rationale

Three load-point options were available (CONTEXT flagged all three):

1. Import from `Base.astro` → global site-wide. Rejected: bloats marketing page + /docs/* with 120 lines of CSS they don't use.
2. Import from `[...slug].astro` directly → route-local. Rejected: if a future surface ever re-uses `PostLayout.astro` (e.g. preview pages, admin renderer), styling would silently drop.
3. **Import from `PostLayout.astro`** → component-local. **Chosen.** Styling travels with the wrapper that needs it, mirroring `Base.astro`'s `global.css` import pattern at one level deeper. Any page that wraps `<PostLayout>` pulls the CSS automatically.

Selector whitelist is the CONTEXT-locked set (9 elements). Explicitly excluded:
- `pre`/`code` — `astro-expressive-code` owns these on `/posts/*` (registered in `astro.config.mjs` Plan 01). Adding `.prose pre` rules here would fight the `.expressive-code` wrapper's own styling and produce visual noise.
- `table` — not in the 9-element whitelist. Tables in posts will render with browser defaults until the first real post needs them; CONTEXT defers this deliberately.
- `strong`/`em` — browser defaults + inherited colour from `.prose` handle emphasis adequately.

Token usage: 18 `var(--*)` references across the file; zero hardcoded hex values. Font sizes and border-radius use absolute units (`rem`, `px`) — that's standard for typography scales and matches `Hero.astro`'s approach.
