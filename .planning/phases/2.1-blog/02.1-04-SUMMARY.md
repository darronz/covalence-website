## Key Technical Decisions

### Why Base.astro alone doesn't cover /docs/*

Starlight's page shell is self-contained — it does not import or extend `src/layouts/Base.astro`. Checking `dist/docs/getting-started/index.html` before the deviation fix showed every non-Starlight head tag present (`canonical`, `og:*`, `twitter:*`, `inter-latin-*` preloads) was absent; Starlight emits its own curated head tag list via `@astrojs/starlight/components` + auto-wired canonical via `astrojs:sitemap`.

The Starlight integration accepts a `head: [{tag, attrs}]` option that injects arbitrary elements into every `/docs/*` page's `<head>`. Adding:

```js
head: [
  {
    tag: 'link',
    attrs: {
      rel: 'alternate',
      type: 'application/rss+xml',
      title: 'Covalence Blog',
      href: '/posts/rss.xml',
    },
  },
],
```

into the `starlight(...)` config closes the gap. The alternate rel tag now ships from BOTH Base.astro (for `/`, `/releases/`, `/posts/*`, `/404`) AND Starlight's head option (for `/docs/*`). No tag is rendered twice on any single page — they're disjoint surfaces.

**Future pattern lock-in:** any future site-wide head tag (e.g. a verification token, a different feed reference, JSON-LD, or a canonical-generation override) must be added in both places if it's meant to reach every route. Flagged for Phase 3 (SEO pass) — a single head-config module that both Base.astro and Starlight consume would DRY this up, but only if we end up with more than one tag that needs to be mirrored. Today's one tag isn't enough to justify that refactor.

### head-extra slot placement

The slot sits immediately before `</head>`, AFTER the font preloads. Intent: per-page head injections can only APPEND, never INJECT ABOVE the base head. This means:
- A post's `og:type=article` coexists with Base.astro's `og:type=website`; social scrapers use the last-seen value (and thus see `article` on post pages, `website` on other pages).
- A post's `<link rel="canonical">` coexists with Base.astro's base canonical — both are technically tolerated by crawlers; the post-specific canonical takes precedence because it appears later and matches the actual URL.

No conflict-resolution logic needed in Base.astro or `[...slug].astro`; the HTML head's latest-value-wins convention is doing the work.

### Empty-collection band gate at the section level

Three options were considered for the `latestPosts.length === 0` case:
1. Render the section with eyebrow + heading and a "No posts yet" line — rejected per CONTEXT voice constraint.
2. Render the section with eyebrow + heading and an empty PostList (PostList itself renders nothing on empty array) — still leaves the eyebrow + heading as orphan cosmetic cruft.
3. **Guard the entire `<section>` with `{latestPosts.length > 0 && …}`** — chosen. Zero visual trace until real posts exist.

Verified against the current empty collection: `grep -c 'Latest writing' dist/index.html` returns 0. The moment the first real post lands, the band lights up with no code change.

### Grep-based acceptance verification and line-counts

First-pass verification using `grep -c 'href="/posts/"' dist/index.html` reported `1` — misleading, because Astro's minifier puts multiple `<a>` tags on one line. `grep -oE 'href="/posts/"' dist/index.html | wc -l` correctly reports `2`. Flagging so Plan 05's CF Pages preview verification (and any future smoke scripts) uses `grep -o | wc -l` when counting occurrences within minified HTML — not `grep -c`.

## Deviations from Plan

- **Found during:** Task 4 post-build verification — `grep -c '<link rel="alternate" type="application/rss+xml"' dist/docs/getting-started/index.html` returned `0`.

- **Files modified:** `astro.config.mjs` (+11 lines).

- **Committed in:** `368d9a3` (separate `fix(02.1-04):` commit, intentionally isolated from the Task 3 wire-up commit so `git bisect` can pinpoint the Starlight head-injection change specifically).
