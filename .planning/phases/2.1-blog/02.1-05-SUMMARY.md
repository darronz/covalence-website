## Regression Detail — Surface E

**Shape:** `/docs/*` code blocks render as plain monospace text — no syntax colouring at all. (Reviewer's direct quote: "E - regression, everything else passed" → clarification: "Plain monospace — no syntax colouring at all".)

**Scope:** Not fully enumerated — reviewer confirmed at least one `/docs/*` page regressed; full-page sweep not completed before this blocker was raised. Gap closure should treat it as systemic until proven otherwise.

**Why this matters:** BRIEF.md line 102 flagged this as the #1 collision risk for the phase: "Starlight (v0.38.3) bundles its own Expressive Code internally. Standalone `astro-expressive-code` install is supposed to supersede Starlight's bundled one per Starlight docs — verify in a preview build that `/docs/*` code blocks still render correctly after the integration is added." Plan 01's executor noted (separately, in `02.1-01-SUMMARY.md`) that Starlight v0.38.3 does NOT auto-defer to the standalone install as BRIEF claimed — the two integrations were observed coexisting side-by-side locally, with Starlight owning `/docs/*` and the standalone owning `/posts/*`. Local `dist/docs/getting-started/index.html` carried `class="expressive-code"` + `ec-line` markup after every wave. In the CF Pages preview, that markup is either absent or not receiving the styling that makes it visible — hence plain monospace.

**Divergence between local build and CF Pages preview:** The most interesting signal. Local `npm run build` produced Expressive Code markup on `/docs/*`; CF Pages preview does not render it. Possible causes for the gap planner to investigate:

1. **Integration-order regression only visible at production-style build.** `astro-expressive-code` is registered BEFORE `starlight()` in the integrations array. Starlight may emit a runtime shim locally but skip it on CF Pages, or vice versa — worth checking CF Pages build logs for warnings or "skipped integration" messages.
2. **Plan 04's `astro.config.mjs` edit (`368d9a3`) adding `head: [...]` to Starlight's integration options.** This was committed to inject an RSS alternate link into `/docs/*`. Check whether touching that options object inadvertently disabled Starlight's internal Expressive Code config or displaced a default.
3. **Theme-name mismatch propagating further than expected.** Wave 1's fix commit (`45c30bb`) swapped the standalone's `themes` to `['night-owl', 'night-owl-light']` because `starlight-dark` / `starlight-light` are Starlight-internal aliases. If the standalone is intercepting `/docs/*` markdown AND Starlight's theme resolution has been displaced, the `/docs/*` blocks would get the standalone's `night-owl` theme at best — OR nothing if the handover is only half-wired. Plain monospace is consistent with the "half-wired" case.
4. **Node version / lockfile drift between local (pinned 22.16.0 via .nvmrc) and CF Pages build runner.** Low probability given CF Pages respects `.nvmrc`, but worth a 30-second check.
5. **Starlight's own Expressive Code pipeline silently disabled once a standalone integration is present at all.** If true, the ONLY path to code blocks on `/docs/*` is via the standalone — which requires Starlight content to flow through the standalone Markdown renderer. That routing is normally implicit; if Plan 04's changes disrupted it, we'd see exactly this symptom.
