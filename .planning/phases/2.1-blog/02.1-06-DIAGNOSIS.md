## Key insight the original diagnosis missed

Both Plan 01 and Plan 04 post-build checks verified only that `class="expressive-code"` and `ec-line` markup were present in the emitted HTML — both counted 32 occurrences, and called it green. That test is necessary but NOT sufficient. Expressive Code's syntax highlighting is expressed via CSS theme selectors in the generated `dist/_assets/ec.<hash>.css` file, and those selectors must match the actual `data-theme` attribute set on the rendered `<html>` element. The Wave 5 reviewer's visual check on CF Pages was the first actual test of whether styling applies; the "CF-Pages-only failure" framing in `02.1-05-SUMMARY.md` is therefore misleading — the regression has been present since Plan 01 landed; it's only visible to a human looking at rendered colour, not to grep-count of class names.

## Why Plan 01's "two EC instances coexist cleanly" was incorrect

Plan 01 concluded the two integrations coexist based on finding `class="expressive-code"` markup in `dist/docs/getting-started/index.html` after registering the standalone. That test only shows the rehype plugin ran — it does not test whether the theme selectors in the generated CSS match the `data-theme` attribute set by Starlight's layout. The generated CSS hash and selectors tell the real story: only ONE `ec.<hash>.css` file ships, and it's built from the first `astroExpressiveCode()` call Astro processes — which is the user-level standalone. Starlight calls `astroExpressiveCode()` again internally at `@astrojs/starlight/integrations/expressive-code/index.ts:108`, but Astro appears to dedupe based on integration name (`astro-expressive-code`) and skip the second registration, so Starlight's `customConfigPreprocessors` never take effect on the CSS generation.

This supersedes the Plan 01 `patterns-established` claim "Standalone Expressive Code coexists with Starlight's bundled EC." The accurate pattern is: **only ONE `astro-expressive-code` integration registration wins, and the first one (the user-level standalone) runs with its own themes instead of Starlight's theme aliases.**

## Remediation chosen — H5-A (remove standalone EC, let Starlight's bundled EC own everything)

Per Plan 06 Task 2 Option H5-A: remove the standalone `expressiveCode({...})` entry from the `integrations:` array. Starlight's bundled EC then owns both `/docs/*` and any other Markdown in the project (including `/posts/*` content), with its native `[data-theme='dark']` / `[data-theme='light']` theme selectors that correctly match the HTML's `data-theme` attribute.

**Wave 1 decision reversal:** Plan 01's Decision 1 ("Use Shiki-bundled `night-owl` + `night-owl-light` themes for the standalone Expressive Code integration") is reversed because the standalone is removed. `/posts/*` code blocks now render with Starlight's `starlight-dark`/`starlight-light` Night-Owl variants — visually near-identical to the BRIEF's originally requested themes and to `/docs/*` themes (exactly what the BRIEF asked for: "visually match /docs/* code blocks").

**Task 3 verifies** that Starlight's bundled EC processes non-docs Markdown (`/posts/*`) via the throwaway diagnostic post at `src/content/posts/2026-04-21-diagnostic-test.md`. If that test had returned 0 EC markup on `/posts/*`, Option H5-B (keep standalone, find a coexistence option) would have been the fallback.

Wave 4's RSS alternate link injection on `/docs/*` (commit `368d9a3`) stays untouched — the Starlight `head:` option is orthogonal to the EC pipeline, as H2 proved.

## Remediation NOT chosen — why not

- **H1 / reorder integrations** — Not applicable once the standalone is removed. The ordering-hygiene pattern established in Plan 01 is kept in spirit (EC-first) but there is now no second EC integration to order against.
- **H2-A / head-alternative plugin or H2-B / Head component override** — Neither applies; H2 is ruled out so the `head:` block stays as it is.
- **H5-B / keep the standalone but suppress its theme registration** — Skipped because H5-A is smaller, fully satisfies the BRIEF, and the throwaway test confirmed Starlight's bundled EC handles `/posts/*` without the standalone.
