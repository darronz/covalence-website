## Key Technical Decisions

### `head-extra` slot: forward-reference, not fallback

The plan flagged an uncertainty in Task 2: `Base.astro` does not currently declare a `<slot name="head-extra" />` in its head block (Plan 04 adds that). The plan specified `<Fragment slot="head-extra">` with a fallback: if the build rejected the unknown slot, emit the metas inside `<PostLayout>` body instead.

**The fallback was not needed.** Astro's documented behaviour is to silently drop children targeting an undefined named slot — no warning, no error, no DOM leakage. Verified empirically: build exits 0 cleanly with the `<Fragment slot="head-extra">` present and Base.astro unchanged. The per-post metas compile today but render nowhere; the moment Plan 04 adds `<slot name="head-extra" />` to Base.astro's head, these tags light up automatically with no changes to `[...slug].astro`.

This is a clean forward-reference pattern — it keeps the route work self-contained in Plan 03 without blocking on a Base.astro edit that belongs in Plan 04's scope.

### RSS body rendering: `marked.parse(post.body)` (not Astro container API)

`@astrojs/rss` accepts a string for each item's `content` field and CDATA-wraps it in `<content:encoded>`. Two paths to get rendered HTML:
- `marked.parse(post.body)` — the project's existing markdown-to-HTML path (see `releases.astro:6-8`).
- `experimental_AstroContainer` + `<Content />` — Astro's newer experimental API.

**Chose `marked.parse`** — the `marked` dep is already present, the config (`{ gfm: true }`) is already set site-wide, and using the container API would introduce a second rendering pipeline for exactly the same content surface. `marked` does NOT sanitise, which is the accepted trade-off per threat model item T-02.1-12 (single-committer repo; mitigation is author discipline + CF preview review, not code). If/when the project opens to external contributors, revisit with `sanitize-html` or `DOMPurify` — explicitly out of scope for Phase 2.1.

### og_image fallback policy

`post.data.og_image` is `z.string().optional()` in the Zod schema. Policy:
1. If present → `new URL(post.data.og_image, Astro.site)` — throws synchronously at build time if the string is not a valid relative-to-base URL. Malformed values (e.g. `javascript:alert(1)`) fail the build loudly rather than silently reaching `<meta>`. This is the T-02.1-10 mitigation promised in Plan 01's threat model.
2. If absent → `new URL('/og-image.png', Astro.site)` — the site-default OG image already present in `/public/og-image.png` and used by `Base.astro` lines 30, 38.

Posts without a custom `og_image` share the site-wide OG card, which is the intended fallback per BRIEF.md line 39.

### Slug derivation: `post.id` directly

The BRIEF flagged URL slug derivation as "frontmatter `slug` OR filename minus date prefix". The v1 Zod schema does NOT include a `slug` field (explicitly locked out per CONTEXT.md `<decisions>`), so there is nothing to derive from the frontmatter side.

Astro's `CollectionEntry` provides `post.id` = filename without `.md` extension. For a file `2026-05-12-project-context-sync.md`, `post.id` is `'2026-05-12-project-context-sync'`, producing URL `/posts/2026-05-12-project-context-sync/`.

**The URL therefore carries the date prefix in v1.** This matches the plan's `params: { slug: post.id }` line literally. If real-post URLs feel date-heavy once the first post lands, a future plan can: (a) add a schema `slug` field, or (b) add a filename-to-slug regex in `getStaticPaths` (`post.id.replace(/^\d{4}-\d{2}-\d{2}-/, '')`). Either is a localised change to this one route file. Deliberately NOT done in v1 so the first real post can inform the decision.
