## Findings

### Warnings (Medium severity — 1)

#### WR-01: `new URL()` is a weaker guard on `og_image` than the plan narrative implies

**Severity:** Medium
**File:** `src/pages/posts/[...slug].astro:25-27`
**Issue:** Plan 03's SUMMARY (and T-02.1-10 mitigation rationale) states that `new URL(post.data.og_image, Astro.site)` "throws synchronously at build time if `og_image` is not a valid relative-to-base URL" and that "malformed values (e.g. `javascript:alert(1)`) fail the build loudly rather than silently reaching `<meta>`." This is not quite what `new URL()` does. Verified empirically on Node 22.16.0:

- `new URL('//evil.com/x.png', 'https://covalence.app').href` → `https://evil.com/x.png` (does NOT throw; protocol-relative becomes cross-origin)
- `new URL('https://evil.com/x.png', 'https://covalence.app').href` → `https://evil.com/x.png` (does NOT throw; absolute override)
- `new URL('javascript:alert(1)', 'https://covalence.app').href` → `javascript:alert(1)` (does NOT throw; scheme passes through into the `<meta property="og:image">` href)

The WHATWG URL parser only throws on truly unparseable input (e.g. `new URL('http://')`). In practice, almost any string `og_image` author might type will parse successfully, and values like `javascript:alert(1)` will end up in the emitted HTML. For a single-committer repo with author-controlled frontmatter, this is low real-world risk (it matches the accepted-risk posture for the `marked.parse()` RSS body — T-02.1-12). But the plan narrative over-claims the guard's strength, which matters if the project ever accepts external contributors or post frontmatter flows from any other source.

**Recommendation:** Either (a) revise the T-02.1-10 mitigation description in a follow-up to accurately reflect what `new URL()` protects against (malformed URL *structure*, not malicious *schemes* or cross-origin *hosts*), or (b) add a constrained check alongside the existing `new URL()` — e.g.:

```astro
const ogImageURL = post.data.og_image
  ? (() => {
      const u = new URL(post.data.og_image, Astro.site);
      if (u.origin !== new URL(Astro.site!).origin) {
        throw new Error(`og_image must be same-origin: got ${u.origin} for ${post.id}`);
      }
      return u;
    })()
  : new URL('/og-image.png', Astro.site);
```

Either route keeps Phase 2.1 scope closed today — this is a phrase-accuracy issue, not a live vulnerability, while the collection has zero author-supplied `og_image` values.

#### IN-04: `Updated {date}` has no `<time datetime>` wrapper in `PostLayout.astro`

**Severity:** Info (semantic HTML polish)
**File:** `src/components/PostLayout.astro:31-36`
**Issue:** The `pub_date` is correctly wrapped in `<time datetime={post.data.pub_date.toISOString()}>`, but the optional `updated_date` is rendered as plain text inside `<span>Updated {updatedFormatted}</span>`. Machine readers (search, Atom-aware crawlers, accessibility tools) lose the ISO timestamp for updates.
**Fix:**
```astro
{updatedFormatted && (
  <>
    <span class="post-meta-sep"> · </span>
    <span>Updated <time datetime={post.data.updated_date!.toISOString()}>{updatedFormatted}</time></span>
  </>
)}
```
The `!` non-null assertion is safe because `updatedFormatted` is truthy iff `post.data.updated_date` is present. Landing this change now avoids a retrofit once a post actually sets `updated_date`.

#### IN-05: RSS feed does not surface `updated_date` nor per-post `og_image`

**Severity:** Info (feature completeness)
**File:** `src/pages/posts/rss.xml.ts:20-37`
**Issue:** `rss.xml.ts` maps to `{ title, description, pubDate, link, content, author }` only. Posts with a non-null `updated_date` do not propagate that into the feed (no `<atom:updated>` element, no `<lastBuildDate>` hint at the channel level), and `og_image` is never exposed via `<enclosure>` or `media:content`. This is fine as an intentional v1 scope decision, but there is no in-code or in-summary note stating it's intentional — it could easily be misread as an oversight by a future reader.
**Fix:** Either (a) add a one-line comment above the item map: `// v1 RSS shape: no <atom:updated> or <enclosure>; author-discipline-gated metadata only. Revisit when cadence warrants.`, or (b) push to Phase 3's SEO sweep if that's already the staged owner of per-post structured data / feed richness. No functional change required today.

#### IN-06: RSS `link` field is a path, not an absolute URL

**Severity:** Info (feed spec pedantry)
**File:** `src/pages/posts/rss.xml.ts:33`
**Issue:** `link: \`/posts/${post.id}/\`` passes a root-relative path. `@astrojs/rss` correctly prepends `site` (passed via `context.site`) when emitting the final XML, so the output is spec-compliant (`<link>https://covalence.app/posts/slug/</link>`). However, the pattern is mildly inconsistent with the `[...slug].astro` convention of using `new URL('/posts/${post.id}/', Astro.site)` for canonicals. Either build absolute URLs in both places (tighter) or leave this as-is and rely on `@astrojs/rss`'s absolutisation (currently fine). Not a bug — just a stylistic inconsistency worth noting if the repo adopts a "always-absolute" rule for future structured data / JSON-LD work in Phase 3.
**Fix:** Optional. If absolutising: `link: new URL(\`/posts/${post.id}/\`, context.site).href`.

## Accepted Risks

These are documented in the phase's threat model and/or SUMMARY files. They are deliberate and should NOT be treated as new findings — they are surfaced here so the next phase's reviewer has them in one place.

Revisit with `sanitize-html` or `DOMPurify` if/when external contributors land. Try/catch is present (verified at `rss.xml.ts:22-27`); a malformed body cannot break sibling items.
2. **T-02.1-10 — `og_image` validation is weaker than the plan narrative stated.** See WR-01 above for the technical nuance; in accepted form, the risk sits at the same author-discipline threshold as T-02.1-12.
3. **ChatGPT MCP-support claim in `getting-started.md:65`** is inaccurate as of 2026-04 but is explicitly deferred to a separate scoped todo (`deb117b`, `b9c86b2`). Per the review context, this is NOT flagged as a bug.
4. **Info-level Astro log line `The collection "posts" does not exist or is empty.`** appears multiple times per build (once per page that calls `getCollection('posts')`). Not a warning, not an error; disappears on first-post publish. Documented in Plans 03 / 04 / 08 SUMMARY files.
5. **Two `getCollection('posts')` calls per page render** (via `Nav` and `Footer` frontmatter in Wave 8). Build time measured at ~1.75s total; overhead is sub-100ms across 11 pages. Astro's collection loader caches; not optimising in v1.
