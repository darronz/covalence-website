# Phase 2.1 Plan 06: Gap-Closure for Surface E Regression Summary

Fix: remove the standalone integration; Starlight's bundled EC now owns /docs/* AND /posts/*, emitting a single `ec.<hash>.css` with `[data-theme='dark'/'light']` selectors that correctly apply.**

## Decisions Made

### Why H5-A over H5-B

H5-A (remove standalone) vs H5-B (keep standalone + find a coexistence option) — H5-A is smaller (3 LOC removed, 0 added), fully satisfies the BRIEF acceptance criterion 2 ("visually match /docs/* code blocks") because both surfaces now use the same Starlight theme set, and was directly verified safe via the throwaway-post test (/posts/* code fences rendered with EC chrome via Starlight's bundled pipeline without any additional scoping). H5-B would have required hunting through `astro-expressive-code`'s typings for a suppression option that may or may not exist for version `^0.41.7`, and maintaining a second EC integration forever just to preserve a theme-choice decision (Plan 01) that's actually moot — because the BRIEF wanted visual parity in the first place.

### Why Wave 1's theme choice reversal is safe

Plan 01 chose `night-owl`/`night-owl-light` (Shiki-bundled) because the standalone integration crashed on `starlight-dark`/`starlight-light` (Starlight-internal aliases). With the standalone removed, that constraint disappears — Starlight's bundled EC consumes its own aliases correctly via its internal theming module (`@astrojs/starlight/integrations/expressive-code/theming.ts`). No theme name is explicitly set anywhere in `astro.config.mjs` now; Starlight uses its defaults. This matches the BRIEF's original intent ("visually match /docs/*") better than Plan 01's workaround ever did.

### Why Wave 4's head-block stays

H2 test definitively showed the `head: [...]` Starlight option is orthogonal to the EC pipeline — removing it did not change the ec.*.css hash, contents, or the emitted HTML's EC markup count. Plan 04's commit `368d9a3` stays in place and all 6 page types continue to carry the RSS alternate link (verified: count=1 on `dist/docs/getting-started/index.html`).

### Why grep-markup-only verification is insufficient

The regression was visible locally from the very first Plan 01 build — grepping for `class="expressive-code"` always returned non-zero because the rehype plugin ran correctly. The actual styling failure was encoded in `dist/_assets/ec.<hash>.css`'s theme selectors, which no Plan 01-04 verification script inspected. This plan's `patterns-established` adds the CSS-selector check as a permanent smoke-test step so this class of regression is caught inside local build verification rather than requiring a human browser session.
