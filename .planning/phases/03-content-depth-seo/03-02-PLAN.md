### Section 1: `## The embedding model`

Cover:
- Model: `nomic-embed-text-v1.5`, fp16 safetensors, ~261 MB bundled inside the app.
- Runtime: `jkrukowski/swift-embeddings` via `MLTensor` on Apple Silicon. Requires macOS 15 (Sequoia) or later. Scheduled across ANE / GPU / CPU by the unified compute stack — not pinned to the Neural Engine.
- Output: a 768-dimension vector. (Flag that the dim is about to shrink — sets up Section 3.)
- Chunking: memories under 512 tokens embed as one chunk; longer memories split into overlapping chunks with 256-token overlap. Search returns the parent note, ranked by its best chunk.

Do NOT write "CoreML on Apple Neural Engine". Pitfall 7.

### Section 2: `## Asymmetric search`

Cover:
- Same model, different prompts. The Nomic embedding family is trained to encode queries and documents asymmetrically — a query embedding and a document embedding of the same text are NOT the same vector.
- The concrete mechanism: two task-specific string prefixes inserted before encoding:
  - `search_document: ` — used when a memory is stored.
  - `search_query: ` — used when a search runs.
- Why it matters: asymmetric encoding puts queries and documents in different regions of vector space, so cosine similarity scores the *retrieval* relationship, not the surface similarity.
- Tone/credibility: one concrete sentence tying this to Covalence's choice ("We could have trained a bi-encoder ourselves. Nomic already shipped one."). Optional — keep if it helps voice, drop if it reads as filler.

### Section 3: `## Matryoshka truncation`

Cover:
- Why Matryoshka: the model was trained so the first N dimensions of its output are themselves a valid (if lower-quality) embedding — Matryoshka Representation Learning, 2022. That means you can truncate without retraining and keep most of the retrieval quality.
- What Covalence does: take the first **256** floats of the 768-dimensional output, then L2-normalize with vDSP (`vDSP_svesq` + `vDSP_vsdiv`). Single truncation — not a multi-stage schedule.
- Trade-off, concretely: "About 2% quality loss, three times smaller vectors — 1 KB per memory instead of 3." Source: code docstring at `EmbeddingEngine.swift:31-33`.
- Denial-cascade beat encouraged here: "No schedule. No rebuild step. Just the first 256 floats and a normalization."

### Section 4: `## Hybrid search`

Cover:
- Two independent queries, merged:
  - **Vector search:** vec0 KNN over the `memories_vec` virtual table (sqlite-vec). Returns nearest neighbours by cosine on the 256-dim vectors.
  - **Keyword search:** FTS5 BM25 over `memories_fts`.
- Fetch limit: `max(limit × 4, 20)` in both branches; ×6 when any filter (tags / source / core) is active. The multiplier exists because filters run *after* fetch in Swift, not in SQL — so the branch must over-fetch to leave enough survivors.
- Pre-RRF threshold: vec0 candidates are filtered by a cosine-similarity minimum (default `0.3`, configurable) before they reach RRF.
- Merge: Reciprocal Rank Fusion. One sentence introducing RRF to a reader who might not know it — Cormack, Clarke, and Büttcher, 2009. Formula: each document's score is `sum over branches of 1 / (k + rank)`. Covalence uses `k = 60`, the widely-cited default from the original paper — the same constant Microsoft Azure and Elastic pick.
- Credibility note worth owning: "RRF k=60 is not our choice — it's the standard."

Optional code snippet (pure illustration, not what the actual Swift looks like):
```text
rrfScore(d) = Σ over branches  1 / (60 + rank(d, branch))
```
If you use a fenced code block here, it MUST start at column 1 (no leading spaces — Pitfall 5 / landmine L2). Language tag `text` is safe.

### Section 5: `## Recency weighting`

Cover:
- After the RRF score is computed, each candidate gets a recency boost based on how old the memory is.
- Function: `decayFactor = 1.0 / (1.0 + ageHours / 8760.0)`. Hyperbolic decay, not exponential. `8760` is hours per year (24 × 365), so a memory exactly one year old gets a decay factor of 0.5 — half-weight in the recency component. Two-year-old memory gets 1/3. Day-old memory gets ~0.997.
- Composition into the final score: `finalScore = 0.9 × rrfScore + 0.1 × decayFactor × 0.033`. The `× 0.033` scales the decay factor into the same numeric range as the RRF score (which sits around 0.03 for a first-rank result with k=60). The `0.1 / 0.9` split means recency contributes **10%** of the final score.
- Design decision worth owning (pulled straight from 03-RESEARCH.md §Tone/credibility notes): "Recency is 10% so that a very recent memory ranks above an equally-relevant old one, without letting recency override genuine relevance."
