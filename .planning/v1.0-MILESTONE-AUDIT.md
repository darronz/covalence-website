## Cross-Phase Integration

### INT-01: Skip-link dead anchor on blog pages (WARNING)

Phase 5 added a skip-link to `Base.astro` targeting `#main-content`. The `id="main-content"` attribute was added to `<main>` in `index.astro` and `releases.astro`, but NOT to the blog pages created by Phase 2.1:

- `src/pages/posts/index.astro` — `<main class="posts-page">` missing `id="main-content"`
- `src/components/PostLayout.astro` — `<main class="post-page">` missing `id="main-content"`

**Mitigation:** Blog nav link is gated on `hasPosts` (currently 0 posts), so no real user encounters the dead anchor today. Fix when first blog post ships.

**Affected requirements:** A11Y-01, A11Y-02 (partial — the skip-link works on the two pages axe-core scanned)

### Dismissed Findings

**RSS autodiscovery ungated (integration checker Finding 2):** INTENTIONAL per decision 02.1-08 — "Gate VISIBLE `<a>` anchors on `hasPosts`, leave INVISIBLE head-level `<link rel="alternate">` unconditional." RSS 2.0 explicitly allows empty channels. Not a gap.

**Standalone astro-expressive-code not installed (integration checker Finding 3):** INCORRECT conclusion. Phase 2.1 Wave 6 intentionally removed the standalone integration because it conflicted with Starlight's bundled EC. Starlight's EC pipeline processes ALL content collections (docs AND posts) via the shared remark/rehype pipeline. Blog post code blocks WILL receive Expressive Code treatment when posts are added.
